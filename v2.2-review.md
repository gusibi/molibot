# v2.2 改进计划评审

**日期：** 2026-05-28  
**评审对象：** `v2.2.md`  
**结论：** 方向合理，但执行计划需要进一步收窄，尤其要明确 Workspace ID 不接管现有目录，并将 ACP 下线改为先删除 active references、暂不物理删除 ACP lib。

---

## 1. 总体判断

`v2.2.md` 相比 `v2.0` 更务实，也比 `v2.1` 更完整。它正确抓住了当前 Agent 主链路的几个核心问题：

1. `runner.ts` 过大，承担了 turn 生命周期、prompt、memory、tool、approval、sandbox、subagent 等过多职责。
2. 工具执行和审批路径分散，Host Bash、Sandbox、Subagent、MCP 等能力缺少统一审计和策略边界。
3. ACP 已经不适合作为 Molibot 当前个人 Agent Runtime 的主边界，应由 Workspace 概念替代。
4. Channel 层应继续只负责消息收发、平台适配和原始消息转换，不应承载队列、恢复、任务编排、工具治理等共享上层能力。

因此，`v2.2` 可以作为后续架构方向文档。但它当前的 Phase 拆分仍然偏大，若直接照做，容易从“渐进改造”退回成一次性大重构。

---

## 2. Workspace ID 与现有目录兼容性

### 2.1 当前结论

Workspace ID 与当前目录逻辑兼容，前提是：

> **Workspace ID 只作为运行身份、权限边界、工具/记忆隔离和审计维度，不接管、不迁移、不重命名当前 `workspaceDir` / `chatDir` / `scratch` / `attachments` / memory 目录结构。**

当前代码里默认 Workspace ID 已经是：

```text
personal
```

因此 `v2.2.md` 中提到的默认 workspace `default` 应统一改为 `personal`。

### 2.2 不建议修改现有目录

当前目录结构没有必要调整。建议继续保留现有 channel/bot/profile 维度的真实文件目录：

```text
data/
  web/
    bots/
      <profileId>/
        <chatId>/
          contexts/
          scratch/
          attachments/
          log.jsonl
          run-summaries.jsonl
          active_session.txt
        MEMORY.md
        skills/

  moli-t/
    bots/
      <telegramBotId>/
        <chatId>/
          contexts/
          scratch/
          attachments/
          ...

  moli-f/
    bots/
      <feishuBotId>/
        <chatId>/
          ...

  moli-q/
    bots/
      <qqBotId>/
        <chatId>/
          ...
```

新增的 Workspace 层不应该变成文件夹，而是进入 SQLite 和运行元数据：

```text
settings.sqlite
  workspaces
    id: personal
    name: Personal
    memory_scope: workspace
```

运行记录增加：

```json
{
  "runId": "...",
  "sessionId": "...",
  "workspaceId": "personal",
  "workspaceDir": "仍然是当前 channel/bot/profile 的实际目录"
}
```

### 2.3 建议写入 v2.2 的边界

建议在 `v2.2.md` 的 Workspace 章节增加明确约束：

```text
Workspace ID 不等于 filesystem workspaceDir。
Phase 1 只建立 workspaceId 元数据和策略边界，不改变现有文件落点。
Workspace.rootPath 第一阶段为 reserved 字段，不参与 runner workspaceDir 选择。
```

这能避免后续实现时误把 Workspace 引入理解成目录迁移。

## 3. ACP 下线策略

### 3.1 当前问题

`v2.2.md` 的 Phase 1 写了：

```text
删除 src/lib/server/acp/ 下的所有文件
```

这个动作过于激进。当前 ACP 虽然已经从 active runtime path 下线，但代码里仍可能存在 settings schema、默认配置、设置页、channel 类型、inactive command 提示等编译期引用。

如果 Phase 1 直接删除 ACP lib，容易导致编译面爆炸，也会让风险集中到一个大变更里。

### 3.2 建议方案

Phase 1 应改为：

```text
下线 ACP active references，暂不物理删除 ACP lib。
```

具体做法：

1. 保留 `src/lib/server/acp/*` 源码，作为 inactive legacy code。
2. 删除或隔离主动入口：
   - Channel runtime 不再启动 ACP service。
   - `/acp`、`/approve`、`/deny` 只返回 inactive 提示。
   - Settings 导航不再展示 ACP 页面。
   - runtime settings 不再依赖 ACP enabled/targets/projects 参与主流程判断。
3. 清理编译期强引用：
   - `settings/defaults.ts`
   - `settings/store.ts`
   - `settings/schema.ts`
   - channel messaging 中仅为旧类型存在的 ACP import
4. 等 active references 清空并经过一轮完整回归后，再单独考虑物理删除 ACP lib。

### 3.3 验收口径

Phase 1 不应以“目录删除”为验收，而应以“主路径无引用”为验收：

```text
Web / CLI / Telegram / Feishu / Weixin / QQ 基础对话不依赖 ACP。
Channel runtime 不实例化 ACP service。
Settings 主导航不再暴露 ACP 入口。
普通 Agent run 不读取 ACP target/project 作为运行边界。
ACP lib 可以暂存，但只能作为 legacy inactive code。
```

## 4. 主要需要优化的点

### 4.1 统一默认 Workspace ID

`v2.2.md` 应把所有默认 workspace 从 `default` 改为 `personal`。

原因：

1. 当前代码已使用 `personal`。
2. 避免后续出现 `default` 与 `personal` 双默认空间。
3. 减少数据库迁移、run metadata、测试和文档不一致。

### 4.2 Phase 2 不要一口气拆完 runner

`runner.ts` 需要拆，但第一轮不应把行数低于 1500 行作为硬验收。

建议 Phase 2 改成 additive 迁移：

```text
先新增轻量 TurnOrchestrator。
先统一 runId / sessionId / workspaceId / runner invocation。
Web 与 stream 入口先接入。
确认稳定后再迁移 IM 和 CLI。
最后再逐步剥离 Memory、Skill、lock、run event 等职责。
```

这样可以先建立主链路边界，而不是直接重写 `runner.ts`。

### 4.3 ToolRuntime 回滚开关要谨慎

`v2.2.md` 里的 `BYPASS_TOOL_RUNTIME` 与“所有敏感执行必须经过 ToolRuntime”的原则冲突。

建议改成：

```text
ToolRuntime 按工具组灰度接入。
未迁移工具可以继续旧路径，但必须列入迁移清单。
已迁移工具不得通过全局开关绕回旧直调路径。
high / critical 工具禁止 fallback 到旧直调路径。
```

这样既保留渐进迁移能力，也不破坏安全边界。

### 4.4 ApprovalBroker 范围需要拆薄

Phase 3 当前同时包含：

```text
ToolRuntime
ApprovalBroker
HostBashStore 迁移
Debounce
Timeout
Subagent 审批上提
Channel renderApproval
```

这个范围太大。建议拆成：

```text
3A. ToolRuntime 先包 built-in tools，补 run/session/workspace 元数据。
3B. Host Bash approval 映射到 ApprovalScope。
3C. Subagent 审批上提到 parent run。
3D. Debounce / timeout 聚合。
```

每一步都可以单独测试和回退。

### 4.5 runs 与 approval schema 需要补状态

`runs.status` 不建议只包含：

```text
running / completed / failed
```

至少应覆盖当前已有运行语义：

```text
running
waiting_for_approval
completed
failed
aborted
expired
```

`approval_requests` 建议增加索引：

```text
(run_id, status)
(session_id, status)
(workspace_id, capability, status)
(created_at)
```

这些对设置页、审批查询、超时清理和 run detail 追溯都会更稳。

## 5. 建议修订后的实施路径

### Phase 1：ACP active path 下线 + Workspace 元数据固化

目标：

```text
不删除 ACP lib。
不修改现有目录结构。
统一 workspaceId=personal。
确保 run/session metadata 写入 workspaceId。
```

验收：

```text
基础对话不依赖 ACP。
Web / CLI / IM run 可以记录 workspaceId。
当前 workspaceDir / chatDir / scratch / attachments / memory 路径不变。
```

### Phase 2：轻量 TurnOrchestrator

目标：

```text
新增 additive TurnOrchestrator。
先统一 runId / sessionId / workspaceId / runner invocation。
先迁移 Web 和 stream。
```

暂不做：

```text
不一次性搬空 runner.ts。
不立即迁移所有 Memory / Skill / stream / lock 逻辑。
```

### Phase 3：逐步迁移 Channel 入口

迁移顺序建议：

```text
Web -> stream -> shared IM -> Telegram -> CLI
```

每迁一个渠道，都验证：

```text
基础对话
附件
停止/插队/队列
Host Bash 审批等待
run detail
session persistence
```

### Phase 4：ToolRuntime 薄包装

目标：

```text
先包 built-in tools。
补统一 tool event metadata。
不改变 individual tool schema。
不一次性迁移 MCP / subagent / plugin tools。
```

### Phase 5：ApprovalScope 与 Host Bash 兼容迁移

目标：

```text
复用现有 Host Bash 能力。
先把 once / session / persistent 语义映射进新 ApprovalScope。
再逐步加入 workspace scope、Debounce、Timeout 和 Subagent 审批上提。
```

### Phase 6：清理 legacy fallback

目标：

```text
等新路径稳定后，再删除 ACP lib、旧 router、旧 approval 分支和临时迁移开关。
```

---

## 6. 最终建议

`v2.2.md` 的目标架构可以保留，但执行规范建议改为：

```text
保留目标架构。
缩小每个 Phase 的一次性改动。
Workspace ID 不碰现有目录。
ACP Phase 1 只删引用，不删 lib。
ToolRuntime 不提供生产级全局 bypass。
ApprovalBroker 拆成多步迁移。
```

这样能最大限度复用当前已经稳定的 channel/bot/profile 目录、Host Bash 审批、PersistentTaskQueue、Memory 和 runner 行为，同时逐步建立新的主链路边界。
