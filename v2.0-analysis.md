# v2.0 架构方案分析报告

**日期：** 2026-05-27
**分析对象：** v2.0.md 架构优化方案

---

## 📊 总体评价

**核心思路可行，但实现太理想化，需要砍一半复杂度。**

---

## ✅ 真正需要优化的部分（高价值）

| 优化点 | 价值 | 原因 |
|--------|------|------|
| **删除 ACP** | 🔥🔥🔥🔥🔥 | ACP 确实是过度设计，当前代码里 ACP 已经存在但可能用得不多 |
| **引入 Workspace 概念** | 🔥🔥🔥🔥 | 用 Workspace 替代 ACP 的 project/target，更符合 personal agent 定位 |
| **统一 Turn 流程** | 🔥🔥🔥🔥 | 当前不同渠道的消息处理确实可能分散，统一入口是好事 |
| **ToolRuntime 统一** | 🔥🔥🔥 | 工具当前在 `agent/tools/`，但 host bash approval 已经是独立模块，统一治理有价值 |
| **Approval 聚合 + scope** | 🔥🔥🔥🔥 | 当前 approval 确实只有 pending/approved/rejected，缺少 scope 控制，这是真实痛点 |
| **Settings 简化** | 🔥🔥🔥 | 当前 `RuntimeSettings` 确实是巨无霸，拆分成 SQLite 表合理 |

---

## ⚠️ 过度设计的部分（可以砍掉）

| 部分 | 问题 | 建议 |
|------|------|------|
| **13+ 个新模块** | 方案拆得太细了（PolicyEngine, ApprovalBroker, SandboxRuntime, MemoryRuntime, RunlogRuntime...） | 保留核心概念，但合并模块 |
| **7 Phase 路线图** | 太复杂，会拖几个月 | 合并成 2-3 个 Phase |
| **PluginManager 最小化** | 当前已经有 Plugin 系统，再做一个是重复 | 复用现有的 Plugin 系统 |
| **复杂的 Subagent 类型** | Scout/Planner/Worker/Reviewer，当前代码里 subagent 已经工作 | 保持现有 subagent，只加权限限制 |
| **SandboxPolicyEngine** | `toolSandbox.ts` 已经有政策，再拆一层没必要 | 把政策逻辑留在现有模块 |

---

## 🎯 我的精简建议（保留 80% 价值，砍掉 50% 复杂度）

### Phase 1：删除 ACP + 引入 Workspace（1-2 周）

```
删除：
- src/lib/server/acp/ 整个目录
- ACP 相关 settings

新增：
- Workspace 模型（替代 project/target）
- 把权限、skills、tools 挂到 Workspace
```

### Phase 2：统一 Turn + Tool 流程（2-3 周）

```
重构：
- 新增 TurnOrchestrator（统一所有渠道入口）
- 新增 ToolRuntime（统一工具调用 + policy check）
- ApprovalBroker（加 scope 控制 + 聚合）

保留：
- 现有的 Sandbox 设置
- 现有的 HostBashStore
- 现有的 Memory 系统
```

### Phase 3：Settings 拆分（1 周）

```
迁移：
- 把巨型 RuntimeSettings 拆到 SQLite 表
- settings.json 只留 bootstrap
```

**不做：**
- 新的 PluginManager（复用现有的）
- 复杂的 MemoryRuntime（当前够用）
- Subagent 类型拆分（当前够用）

## 🔍 具体分析：哪些是真痛点？

### 真痛点 1：ACP 确实没用

```typescript
// 你当前的 ACP 代码
src/lib/server/acp/
├── service.ts
├── connection.ts
├── providers/index.ts
└── types.ts
```

**分析：** ACP 是为了控制 Claude Code/Codex，但可能你根本没用。删除是对的。

---

### 真痛点 2：Approval 缺少 Scope

```typescript
// 当前的 approval 只有：
type HostToolApprovalStatus = "pending" | "approved" | "rejected";

// 方案加的 scope 很有用：
type ApprovalScope = "once" | "turn" | "session" | "workspace" | "persistent";
```

**分析：** 这是真需要的。用户会烦每次都要批准同一个操作。

---

### 真痛点 3：Settings 太巨型

```typescript
// 当前 RuntimeSettings 包含：
- providerMode, piModelProvider, piModelName
- defaultThinkingLevel
- customProviders, defaultCustomProviderId
- modelRouting, modelFallback
- compaction
- systemPrompt, timezone
- acp (整个 ACP 设置)
- agents (agents 数组)
- channels (channels map)
- mcpServers
- skillSearch, skillDrafts
- toolSandbox (整个沙箱设置)
- hostTools (整个主机工具设置)
- disabledSkillPaths
- telegramBots, feishuBots, qqBots
- plugins
```

**分析：** 拆成 SQLite 表是对的，太多了。

---

### 伪痛点 1：需要 PolicyEngine？

```typescript
// 方案设计：
PolicyEngine -> ApprovalBroker -> SandboxRuntime

// 现实：
你现有的 toolSandbox.ts 已经有政策逻辑，
hostBash/approval.ts 已经有审批流程。
```

**分析：** 把政策逻辑合并到 ToolRuntime 就行，不需要单独拆。

---

### 伪痛点 2：需要 MemoryRuntime？

```typescript
// 你当前的 memory/ 已经有：
- MemoryGateway
- 各种 backend
```

**分析：** 当前够用，不需要大改。

---

### 伪痛点 3：需要 SkillLoader 标准化？

```typescript
// 方案说要扫描：
.pi/skills / .agents/skills / ~/.pi/agent/skills / ~/.agents/skills

// 现实：
你当前已经有 skill 系统了，不需要兼容 Pi 的目录结构。
```

**分析：** 保持你现有的 skill 系统就行。

## 📋 最终建议：3-Phase 精简方案

### Phase 1：删除 ACP + Workspace（1-2 周）

```
删除：
- src/lib/server/acp/
- RuntimeSettings.acp

新增：
- Workspace 模型（SQLite 表）
- workspace_id 关联到 session/run
```

### Phase 2：TurnOrchestrator + ToolRuntime + Approval（2-3 周）

```
新增：
- TurnOrchestrator（统一入口）
- ToolRuntime（统一工具调用）
- ApprovalScope（once/turn/session/workspace/persistent）
- Approval 聚合（debounce + merge）

重构：
- Web/CLI/Telegram/Feishu 都走 TurnOrchestrator
- 所有工具调用走 ToolRuntime
- HostBashStore 集成 approval scope

保留：
- 现有的 toolSandbox.ts
- 现有的 memory/
- 现有的 plugins/
```

### Phase 3：Settings 拆分（1 周）

```
迁移：
- settings.json 只留 bootstrap
- 其他配置拆到 SQLite 表
```

**总计：4-6 周，而不是方案里的数月。**
