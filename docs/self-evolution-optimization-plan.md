# Molibot 参考 Hermes 的自我进化优化方案

日期：2026-04-11
状态：待评审，暂不改代码
参考来源：
- `~/Github/molipibot/docs/hermes-agent.md`
- `~/Github/hermes-agent`

## 目标

参考 Hermes 的设计思路，补齐 Molibot 当前实现里“会越用越稳、越做越会”的关键缺口，但不直接照搬 Hermes 的全部体系。

本轮目标不是引入训练系统，也不是做大规模重构，而是优先补齐运行时自我进化能力：

1. 让任务执行更稳，有明确边界
2. 让成功经验能沉淀成可复用技能
3. 让记忆真正服务当前决策，而不是单纯堆积
4. 让每次任务结束后有结构化复盘和可观察记录

## 当前现状判断

Molibot 当前已经具备以下基础：

- 多渠道统一运行骨架
- 技能目录与技能路由
- 记忆网关与 Mory 后端
- Prompt 模块化组装
- 工具优先级和调用约束
- MCP 按需暴露

但距离“自我进化 agent”还缺少几个核心环节：

### 1. 只会使用技能，不会沉淀技能

当前技能主要能力是：
- 加载
- 列表展示
- 显式匹配
- prompt 注入

缺少：
- 成功任务后的技能沉淀入口
- 技能更新/淘汰机制
- 技能质量检查和风险扫描
- 技能创建触发标准

### 2. 记忆能存，但没有形成稳定的决策闭环

当前记忆已经支持：
- gateway 统一入口
- layered memory
- hybrid search
- Mory 后端

缺少：
- 会话级稳定快照策略
- 明确的记忆写入门槛
- 使用旧记忆前的现实校验流程
- 针对“当前任务”与“长期知识”的更强分层

### 3. 缺少任务完成后的复盘层

当前运行结束后没有统一的：
- 本次任务是否成功
- 关键路径是什么
- 哪一步失败过
- 哪些做法值得复用
- 哪些坑应该避免

### 4. 缺少运行边界控制

相比 Hermes，当前还缺少更明确的：
- 最大轮次预算
- 连续失败止损
- 分类型重试/降级
- 任务阶段状态

## 从 Hermes 值得借鉴的部分

建议直接参考，但按 Molibot 现状做裁剪：

### A. 技能 = 程序性记忆

Hermes 最值得借的点，不是“技能目录”，而是：
- 技能代表“某类任务怎么做”
- 成功后可以沉淀
- 下次按需加载
- 主 prompt 里只放索引，不放全文

Molibot 应该补齐：
- 技能管理入口
- 技能沉淀策略
- 技能更新策略
- 技能安全检查

### B. 记忆分层并且受控注入

Hermes 的思路更稳：
- 长期事实和用户偏好分开
- 会话开始时形成稳定快照
- 中途更新不等于立刻污染本轮判断

Molibot 应该补齐：
- prompt 记忆注入的快照化
- 写入时分类和门槛
- 过期/冲突记忆的处理规则

### C. 任务有预算，不无限探索

Hermes 的预算机制非常关键。
Molibot 也应该对每次运行增加：
- 最大推理/工具轮次
- 连续失败上限
- 工具失败后的降级策略

### D. 任务结束后留下结构化轨迹

Hermes 后面能做更深训练，是因为运行轨迹结构清楚。
Molibot 当前不需要上训练，但应该先把：
- 任务结果
- 工具路径
- 失败点
- 可复用经验

记录下来。

## 不建议现在照搬的部分

以下内容当前阶段先不做：

1. RL / 训练环境
2. 大批量轨迹生成
3. Reward / benchmark 环境
4. 模型级自我强化闭环

原因：
- 投入大
- 对当前产品价值不够直接
- 会显著扩大系统复杂度
- 先把运行时闭环做稳，收益更高

## 建议的整体优化路线

建议分 4 个阶段推进。

---

## 第一阶段：先把运行主干稳住

### 目标

让任务执行过程具备明确边界、稳定节奏和统一收尾。

### 要做的事

1. 为每次任务增加运行预算
   - 最大轮次
   - 连续失败上限
   - 工具调用总量上限
   - 特定可退款场景可后续再加

2. 为运行过程定义固定阶段
   - 理解任务
   - 选技能 / 选工具
   - 执行
   - 校验
   - 收尾

3. 为运行结束增加结构化结果对象
   - 是否成功
   - 使用了哪些工具
   - 是否命中技能
   - 是否发生降级
   - 失败点是什么
   - 是否值得沉淀为技能 / 记忆

4. 为失败增加清晰分流
   - 可重试失败
   - 需降级失败
   - 必须停下失败

### 主要改动点

- `src/lib/server/agent/runner.ts`
- `src/lib/server/agent/prompt.ts`
- 可能新增：
  - `src/lib/server/agent/runtimeBudget.ts`
  - `src/lib/server/agent/runSummary.ts`
  - `src/lib/server/agent/failurePolicy.ts`

### 预期收益

- 长任务更稳
- 不再容易陷入重复尝试
- 为后续技能沉淀和复盘提供统一挂点

---

## 第二阶段：补上技能沉淀闭环

### 目标

把一次成功任务沉淀成下次能直接复用的方法。

### 要做的事

1. 新增正式的技能管理能力
   - create
   - update
   - deprecate / archive
   - preview

2. 定义技能沉淀触发条件
   - 任务复杂度达到阈值
   - 成功完成
   - 中间解决过明显障碍
   - 对同类问题具备复用价值

3. 默认不直接自动落盘
   - 先生成“建议保存的技能草稿”
   - 由你 review 后再保存
   - 后面可加开关支持自动保存

4. 统一技能格式
   - 适用场景
   - 触发词 / 识别条件
   - 步骤
   - 常见坑
   - 验证方式
   - 可选依赖（如 MCP）

5. 增加技能安全检查
   - prompt 注入风险
   - 明显危险命令
   - 越权路径写入

### 主要改动点

- `src/lib/server/agent/skills.ts`
- `src/lib/server/agent/tools/index.ts`
- 新增建议：
  - `src/lib/server/agent/tools/skillManage.ts`
  - `src/lib/server/agent/skillDraft.ts`
  - `src/lib/server/agent/skillGuard.ts`

### 预期收益

- Molibot 会开始积累“做事方法”
- 不再每次遇到复杂任务都从头摸索
- 技能从静态文档变成动态经验库

---

## 第三阶段：把记忆升级成决策辅助层

### 目标

让记忆更像“可靠上下文”，而不是“内容仓库”。

### 要做的事

1. 明确记忆类型
   - 用户偏好
   - 项目规则
   - 长期背景
   - 当前阶段任务状态
   - 临时过程记录

2. 写入前先做分类和门槛判断
   - 值不值得记
   - 应该进长期还是短期
   - 是否已有重复项
   - 是否与旧条目冲突

3. prompt 注入改成更受控的快照方式
   - 本轮开始时生成稳定上下文
   - 运行中写入不直接污染当前轮判断
   - 下轮再生效

4. 使用旧记忆前做现实校验
   - 如果与当前文件/设置/运行状态冲突，优先信当前现实
   - 同时提示更新或清除旧记忆

5. 为记忆检索增加任务导向
   - 当前问题是查事实、找偏好、找项目约束，还是找进行中的工作

### 主要改动点

- `src/lib/server/memory/gateway.ts`
- `src/lib/server/memory/moryCore.ts`
- `src/lib/server/agent/tools/memory.ts`
- `src/lib/server/agent/prompt.ts`
- 可能新增：
  - `src/lib/server/memory/memorySnapshot.ts`
  - `src/lib/server/memory/memoryWritePolicy.ts`

### 预期收益

- 记忆更干净
- 记忆更少打扰当前任务
- 旧记忆误导当前判断的风险更低

---

## 第四阶段：补上复盘与可观察性

### 目标

让系统知道自己哪里做得好，哪里做得差，并让人能看见。

### 要做的事

1. 为每次运行记录结构化摘要
   - 用户目标
   - 采用路径
   - 技能命中情况
   - 工具调用概况
   - 失败与降级
   - 最终结果

2. 为成功任务生成经验候选
   - 候选技能
   - 候选记忆
   - 候选规则改进建议

3. 为失败任务生成复盘
   - 卡在哪
   - 是能力缺失还是路由错误
   - 下次应该先尝试什么

4. 后续可选加一个设置页查看这些记录
   - 先不作为第一优先级

### 主要改动点

- `src/lib/server/agent/runner.ts`
- 可能新增：
  - `src/lib/server/agent/reflection.ts`
  - `src/lib/server/agent/runLog.ts`
  - `src/routes/settings/agent-runs/+page.svelte`（后续可选）

### 预期收益

- 更容易判断系统是否真的在进步
- 后续要做更深的自动优化会容易很多

## 推荐实施顺序

建议按下面顺序做：

1. 第一阶段：运行主干稳住
2. 第二阶段：技能沉淀闭环
3. 第三阶段：记忆升级
4. 第四阶段：复盘与可观察性

原因：
- 先稳执行
- 再积累方法
- 再让记忆更可靠
- 最后补长期反馈环

## 第一轮实际建议落地范围

如果准备开始第一轮改动，我建议只做一个“低风险但立刻见效”的版本：

### 第一轮范围

1. 增加任务预算与失败止损
2. 增加统一任务收尾摘要
3. 新增技能管理入口（先支持草稿生成，不强行自动保存）
4. 增加“任务完成后建议保存为技能”的机制
5. 不碰训练系统
6. 不做大 UI 改造

### 为什么这样切

因为这是最容易马上产生价值的一组改动：
- 任务更稳
- 经验开始沉淀
- 不会把改动面一下子铺太大

## 开工前检查清单

在真正改代码前，先确认：

1. 是否按推荐顺序推进
2. 第一轮是否只做“预算 + 收尾 + 技能草稿”
3. 技能沉淀是否默认“先生成草稿，再人工确认保存”
4. 是否要在第一轮就增加运行记录文件
