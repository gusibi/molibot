# Findings

## Confirmed Product Constraints

- Target: Apple Silicon, macOS 13+, unsigned beta first.
- Tauri owns native windows, tray/menu, notifications, and sidecar lifecycle.
- Existing Node/SvelteKit server remains the only Agent/channel runtime and continues using `~/.molibot`.
- Desktop UI is a separate Svelte application; shared code is limited to stable contracts, tokens, i18n primitives, and event schemas.
- Phase 1 must distinguish App-managed and compatible external services; quitting must never stop an external service.

## Repository State

- Root project uses SvelteKit 2, Svelte 5, Vite 7, adapter-node, Node >=22.5.
- Local toolchain includes Node 22.15.1 and Rust/Cargo 1.89.
- The workspace already contains uncommitted design and planning files; they must not be overwritten except the planning-with-files session files.
- `DESIGN.md` and `Momo for Mac (standalone).html` provide the confirmed visual target.

## Open Architecture Questions

- Decide whether to bundle production dependencies as `node_modules` resources for the first beta or deepen the adapter bundle before the DMG size-optimization pass.

## Phase 1 Foundation

- Tauri 2.11 is the current official host baseline; single-instance must be registered before other plugins.
- The official autostart plugin supports macOS LaunchAgent registration and explicit enable/disable permissions.
- The official Node sidecar path requires a target-triple-named executable and `bundle.externalBin`; this remains a later Phase 1 slice because the current server still assumes source/build resources and `process.cwd()`.
- No macOS private API is needed for the initial native window shell.

## Sidecar Packaging Audit

- The canonical supported production path is `npm start` / `molibot start`; both currently execute `node build`. The background service wrapper ultimately runs the same command.
- The adapter-node artifact is self-starting at `build/index.js`, but all root production dependencies remain external. The source checkout currently has a 13 MB build tree and a 533 MB full `node_modules`; the desktop release needs an `npm ci --omit=dev` measurement before the 100 MB DMG target can be evaluated honestly.
- Runtime assets still use release-root-relative paths for `package.json`, the vision smoke fixture, QQ helper scripts, and subagent definitions. Running the sidecar with the packaged release root as `cwd` preserves those assumptions without writing machine-local paths into code.
- A process-level data-directory lease belongs in the shared server launcher so standalone server and desktop startup obey the same single-writer rule. The desktop host must not implement a private second lock that standalone Molibot ignores.
- Handshake can remain a normal localhost read-only endpoint; desktop-only privileged APIs will separately require the ephemeral token passed by the Tauri host.
- The pinned runtime is the official Apple Silicon Node 22.23.1 archive (`node-v22.23.1-darwin-arm64.tar.xz`) with SHA-256 `fb526811860f81dcac7dd8b2b55eca4accfc5d61c3b7c2508f2639faee8a738d`.
- Initial production resources were 416 MB plus a 112 MB Node binary. Moving build-only `pi-web-ui`/`mini-lit`/`lit`/Express packages to dev dependencies and omitting optional packages reduced runtime resources to 249 MB while the packaged handler still initializes successfully.
- The first full bundle produced a valid 530 MB `.app` and a temporary 758 MB read-write DMG, but did not finish the compressed DMG. Live process diagnosis was unavailable after host-approval quota exhaustion; this remains unverified rather than being reported as a successful DMG.
- The generated plist defaulted to macOS 10.13, contradicting the product requirement. Bundle configuration now explicitly sets `minimumSystemVersion` to `13.0`.
- The managed child needs graceful termination because immediate `kill()` can skip server lease cleanup. The supervisor now sends SIGTERM, waits up to five seconds, and only then force-kills; a shell-trap unit test verifies the graceful signal arrives.
- Verbose Tauri bundling showed the remaining DMG failure is `hdiutil: create failed - device not configured` inside the restricted execution environment. Release builds now use Tauri `--ci` and automatically create a `.sha256` after a successful DMG, but the actual image must still be produced and inspected where macOS disk-device access is allowed.

## Phase 2 Desktop Chat Audit

- Existing `/api/sessions`, `/api/sessions/[id]`, `/api/stream`, and `/api/stream/stop` already use `channel=web`, Web Profile identity, the shared session store, and the shared Agent runner. Desktop should consume these contracts rather than add a `desktop` channel or duplicate runtime logic.
- Web Profiles are currently read from `settings.channels.web.instances`; `/api/settings` exposes them but also returns the full settings object. A narrow read-only Desktop bootstrap endpoint is preferable so the Chat shell does not receive unrelated credentials or settings.
- The session detail route supports GET and rename PUT but has no DELETE handler, even though the plan requires local session deletion. This must be added in the shared API/store path, not implemented as Desktop-only file manipulation.
- The Desktop WebView and loopback SvelteKit server are different origins. Direct browser `fetch` would be subject to CORS, so the Desktop needs an explicit Tauri HTTP/transport capability or a native command bridge; hard-coding permissive server CORS is not an acceptable substitute.
- Tauri 2's official HTTP plugin exposes a Fetch-compatible JavaScript API backed by Rust and requires an explicit URL scope. It supports response body reads/cancellation, so it is suitable for the existing SSE stream while restricting Desktop network access to loopback HTTP URLs only.
- A standard-width browser smoke against a temporary DATA_DIR proved the Desktop can load persisted Web sessions, switch transcript, create a session, rename it inline, and delete it with an explicit two-step confirmation. No real channel runtime was started.
- JavaScript prompt/confirm dialogs were removed after the browser could not reliably observe them; sidebar-local edit/confirm states are clearer, fully testable, and better match the macOS list interaction.
- The existing read-only `/api/web/files` endpoint already lists a session's persisted attachments (base64url file id, original name, media type, size, createdAt) and serves each file inline or as a download. Desktop reuses it directly for the current-session file panel instead of adding a desktop-specific file API, keeping the per-session scope server-owned.
- In-app preview and download avoid new native plugins: the loopback Tauri HTTP client fetches bytes and the WebView builds `URL.createObjectURL` blobs for image/audio/video preview and an anchor `download` for export. The HTTP capability must additionally allow `/api/web/files*` on `127.0.0.1`/`localhost`.
- Native "reveal in Finder" and local-cache Quick Look need the file's absolute disk path plus `fs`/`dialog`/`opener` JS plugins (the Rust `tauri-plugin-opener` crate is already declared but not wired on the JS side or in capabilities). They are deferred to a dedicated slice so the read-only panel can ship without exposing absolute workspace paths to the WebView.
- File upload must use the non-streaming `/api/chat` multipart endpoint: `/api/stream` is JSON-only and rejects files, while `/api/chat` already saves attachments, runs the shared Agent turn, and persists both the user message attachments and reply. Desktop therefore branches — file-bearing turns POST multipart to `/api/chat`, plain-text turns keep the `/api/stream` SSE path — instead of duplicating upload handling.
- The `@tauri-apps/plugin-http` `fetch` wraps the request in a standard `Request` and reads `req.arrayBuffer()`, then forwards both the bytes and the `Content-Type` header the `Request` generated. So passing a `FormData` body works through the loopback HTTP client as long as no manual `Content-Type` is set (the multipart boundary must come from the serializer). This is the only CORS-safe way to upload from the WebView; a plain `globalThis.fetch` to the loopback origin would be blocked.
- Host Bash approvals have no dedicated REST endpoint: the `/api/stream` run emits a `host_bash_approval` SSE event (the server-side `HostBashApprovalPrompt`) and stops with `stopReason: "waiting_for_approval"`, and decisions are resolved by posting `/hosttools approve-once|approve-session|approve|reject <requestId>` to `/api/chat`. That command path is intentionally not persisted as a session message, executes the approved command, rewrites the tool result in context, and fires a background `runner.run(...)` to resume — so the resumed answer is appended asynchronously and the Desktop must poll the transcript rather than expecting a streamed continuation. Reusing this shared path (instead of adding a desktop approval API) keeps the control directive out of the transcript per plan §12.
- The Desktop and the `/api/stream`/`/api/chat` calls all use the same default Web `externalUserId` (derived from an empty userId + the active profileId), so an approval registered during a stream run is found when resolved through `/api/chat` with the same profileId and conversationId.
- The desktop CSP in `apps/desktop/src-tauri/tauri.conf.json` controls what the WebView may load. It must include `blob:` in `img-src` and a `media-src 'self' blob:` for the file-panel / inline-attachment / media previews to work, because those use `URL.createObjectURL`. The loopback `/api/...` fetches are unaffected by `connect-src` because they go through the Tauri HTTP plugin (Rust/IPC), not a direct browser `connect`.

## Phase 5 Release Workflow

- The repository had no `.github/workflows/` directory; Phase 5 release infrastructure starts with a dedicated `desktop-release.yml` rather than extending an existing pipeline.
- `npm run desktop:build` (root) is the single reproducible entry point: it runs `apps/desktop`'s `tauri:build`, which is `prepare:runtime && tauri build --ci --config src-tauri/tauri.bundle.conf.json && finalize-desktop-release.mjs`. `prepare-desktop-runtime.mjs` downloads the pinned Node 22.23.1 darwin-arm64 archive (SHA-256 verified), runs the root SvelteKit build, and assembles the runtime dir via `bin/molibot-release.sh` with `MOLIBOT_RELEASE_SKIP_BUILD=1` (which still runs `npm ci --omit=dev --omit=optional` inside the runtime dir). Root deps must be `npm ci`-installed before this, because `prepare:runtime` runs `npm run build` at the root.
- The DMG filename and version are fixed by `apps/desktop/src-tauri/tauri.conf.json` (`Molibot` / `0.1.0`), so the workflow does not override the version from the tag; instead it records the tag, git commit, and build environment in a BUILD-INFO.txt manifest alongside the DMG and `.sha256` (plan §16.1 "同时发布 checksum、版本号、Git commit 和构建环境信息").
- `macos-14` GitHub runners are Apple Silicon, matching the product target; `MACOSX_DEPLOYMENT_TARGET=13.0` is set at the job level to match the bundle's `minimumSystemVersion`. The build is unsigned (no `signingIdentity`, no notarization), consistent with the unsigned-beta scope; the release is marked `prerelease: true` and the manifest states Gatekeeper may require an explicit override.
- The release-publish step is gated on `startsWith(github.ref, 'refs/tags/molibot-v')`, so a `workflow_dispatch` run builds and uploads a workflow artifact without publishing — useful for verifying the DMG pipeline on a real runner without cutting a release. The existing server/Web product uses `v*` tags, so `molibot-v*` keeps desktop releases disjoint.
- End-to-end DMG production still cannot run in the restricted sandbox (`hdiutil`/`tauri build` need real macOS disk-device access); the workflow file is the deliverable and is YAML-validated, but a real `molibot-v*` tag or `workflow_dispatch` run is required to confirm the DMG is produced and the release is published.

## Phase 3 External-Channel Aggregation

- The shared `Conversation` type had no external-channel metadata; plan §7.2 requires bot-instance, sender, chat-type/thread, and platform metadata to live in the shared session layer (not channel code). Added it as a single optional `external?: ExternalSessionMetadata` on `Conversation` so all fields stay backward-compatible and old records simply omit it — channel adapters populate it later when mapping platform fields to the unified structure.
- The legacy session index (`byConversationId`) stores only `{ channel, externalUserId }` per conversation; there is no per-session bot-instance id. So the first aggregation slice groups by channel only. Bot-instance hierarchy ("渠道 → Bot 实例 → 会话", plan §7.2) needs `external.botInstanceId` to be populated and indexed by channel adapters first; that is deferred to a later Phase 3 slice.
- `SessionStore` had no method to enumerate sessions across all external users of a channel (only `listConversations(channel, externalUserId)` for one user). Added `listExternalSessions()` which walks the legacy index, skips `web`, loads each file newest-first, and skips stale entries — the read-only aggregation reads through it rather than touching the index file format directly.
- External sessions carry message content (transcript), which is user data; the list endpoint deliberately loads only `conversation` metadata (title/updatedAt/external), never `messages`. The raw `externalUserId` is a platform chat id — not display-meaningful and not needed for the read-only list — so `maskExternalUserId` truncates it to an 8-char preview rather than passing the full id to the WebView. The full transcript view (a later slice) will load messages per-session on demand.
- `cli` sessions are non-web but local (not an external channel in plan §7.2); the mapper filters to the four known external channels (telegram/feishu/qq/weixin) so cli sessions never appear in the aggregated list.
- The §11 credential-boundary review rule ("build field-by-field, never `return {...source}`") applies here too: `buildDesktopExternalSession` projects each field explicitly and only attaches optional fields (`senderAvatarUrl`/`threadTitle`/`botInstanceName`) when present, so no future-added `Conversation` field can accidentally leak through a spread.
- The ChatView read-only external view is the first Phase 3 surface and is intentionally list-only: it groups by channel and shows display metadata, but does NOT load transcripts. Plan §7.2 requires external transcripts be read-only with no input/rename/delete — the list view already enforces that (no affordances rendered). A later slice adds the read-only transcript pane (loading `/api/sessions/[id]` messages for an external conversation on demand), and §7.3 real-time events will refresh the list/transcript without polling SQLite.
- The Desktop client groups by channel only (matching the server), not by Bot instance, because the legacy session index has no per-session `botInstanceId` yet. The "渠道 → Bot 实例 → 会话" hierarchy (plan §7.2) is deferred until channel adapters populate `external.botInstanceId` and it is indexed; the view list is structured so a later grouping-by-instance pass can nest under each channel without changing the row shape.
- The external transcript is served by a dedicated `/api/desktop/external-sessions/[id]` GET, not the existing `/api/sessions/[id]`. The latter is Web-scoped: it requires a Web `externalUserId` (derived from userId + profileId) and looks up `channel: "web"` only, so it cannot resolve telegram/feishu/qq/weixin sessions. The desktop endpoint resolves via `SessionStore.getExternalSession(id)` against the legacy index instead. This keeps the Web contract unchanged and avoids exposing external sessions through the Web-scoped route.
- External attachments cannot be previewed/downloaded through `/api/web/files` (that endpoint is Web-profile-scoped and keys files by Web session). So the transcript mapper drops the attachment `local` path entirely and keeps only display-safe fields (original name, mediaType, mimeType, size) — the pane shows attachment names as a read-only list, matching plan §7.2's "外部附件在原消息位置展示" without promising preview for external attachments. Native Finder reveal/Quick Look for external attachments would need the absolute disk path and is deferred.
- `system` role messages are control directives (operator profile directives, approval control text). Plan §12 requires temporary control directives not persist as ordinary session messages; even though some may already be stored, the read-only transcript filters them out so they never surface to the desktop user.

## Phase 4 First-Launch Onboarding

- The §9.1 triage is fully derivable from the existing `summarizeDesktopReadiness` (model + Web Profile presence), so no new endpoint is needed. `classifyFirstLaunch` encodes the three-way split: both present → `usable`; neither → `new`; exactly one → `broken`. The "broken" branch is the §9.1 "config exists but unusable → lightweight repair, don't overwrite" case — the overlay only guides (Open Settings) and never writes config, matching the plan.
- The overlay is gated by a localStorage seen-flag (`molibot-desktop-first-launch-seen`), not by a server-side "first run" record. This keeps it desktop-only and avoids touching `~/.molibot`; an existing user who already has a usable config sees the `usable` summary once (then Continue dismisses it), and a genuinely fresh install sees `new`. The flag is shared across the Chat/Settings webviews only if both read it, but the overlay only renders in Chat, so Settings is unaffected.
- The `usable` branch offers "Continue" (dismiss); `new`/`broken` offer "Don't show again" (also dismiss). Both write the same seen-flag — the difference is copy, since a `new`/`broken` user should be nudged toward Settings rather than told everything is ready. Plan §9.1 says a fresh install must not fake a completed setup, so `new` never shows a "Continue to Chat" primary action.
- Full §9.2 guided setup (provider validation, default Agent/Web Profile creation, optional channel connection, login-start toggle, diagnostics) and the §9 migration/health-check summary for existing-but-usable configs are deferred — they need multi-step form UI and a running service to validate provider credentials. The current overlay is the triage entry point that routes into the existing Settings sections.

## Phase 4 Runtime Environment (§10)

- The §10 Runtime environment page is read-only detection/display first; per-item authorized install execution is deferred because it needs native PATH recovery for Finder-launched apps, real-time log streaming, cancel/retry, and a structured install-event record (plan §10 "安装事实记录为不会回灌模型上下文的结构化运行事件"). The current slice delivers the detection contract, the displayed fields, and the exact install command — everything the user needs to see *before* authorizing an install.
- Detection resolves each binary via `command -v` (login-less `sh`), then classifies source from the resolved path (`/opt/homebrew/` or `/homebrew/` → homebrew, else system) and probes the version arg. It is split into an injectable `WhichResolver` so the pure mapper is unit-testable without spawning processes; the route injects the live `defaultWhichResolver`. A probe failure never throws — it yields "missing"/"unknown" so the page always renders.
- Node is intentionally omitted from the dependency list: the bundled Node 22 sidecar already satisfies it (plan §10 "安装包只强制内置 Node 22 sidecar"). The page lists only tools the user may still need to add (ffmpeg, git, python3).
- The §11 credential/path boundary applies: `buildDesktopRuntimeDependency` projects fields explicitly and drops the resolved binary path — only the source *category* ("homebrew"/"system") reaches the WebView, never `/opt/homebrew/bin/ffmpeg` or `/usr/bin/git`. Install commands are the safe forms only: `brew install <formula>` or `pip install --target ~/.molibot/tooling <pkg>` — never `sudo`, `npm -g`, or system-Python modification (plan §10 "禁止 sudo pip、全局 npm -g 和修改系统 Python"). A regression test asserts no declared dep's install command contains `sudo` or `npm -g`.

## Phase 4 Guided Onboarding (§9.2)

- The §9.2 guided flow is structured as an ordered step model (`ONBOARDING_STEPS`: provider → agent → channels → launch → diagnostics) so the overlay can advance/Back/Finish without per-step branching logic. `classifyFirstLaunch` gates whether the guided flow shows: `usable` keeps the simple summary (no steps), `new`/`broken` get the full guided walk. This matches §9.1's "全新安装：完整引导" and "配置存在但不可用：精简修复引导" without overwriting existing config — the guided flow only collects a draft and routes to Settings; it never writes config directly.
- The provider draft is **credential-blind** by design: `ProviderDraft` carries `apiKeyPresent: boolean`, never the key value. The API-key `<input type="password">` updates only the boolean; the key itself lives only in the input element and is never serialized into the draft object or any reactive state that could be logged. This pre-empts the §11.2 "UI、日志、诊断导出和 API 默认掩码" requirement before submit is even wired — when submit arrives, the key will go straight from the input to a desktop-token-gated server route, never through the draft.
- `validateProviderDraft` is structure-only (non-empty name/model, known protocol, http(s) baseUrl, apiKeyPresent). It deliberately does NOT contact the provider — live verification (plan §9.2 "验证") needs the existing `/api/settings/provider-test` path, which currently takes a plaintext key and is not desktop-token-gated. Wiring a desktop-safe verify route is deferred; the form's submit/verify note says so explicitly rather than faking success (§9.2 "没有可用模型时不得假装配置完成").
- The Agent and Launch steps are now implemented in later slices. Channels and Diagnostics remain placeholders because they still need channel-specific connection workflows and a live diagnostics contract; the ordered step model keeps those additions isolated.
- The §9.1 "migration and health-check summary" for existing-but-usable configs is fully derivable from the already-loaded readiness data, so it needed no new endpoint. `buildOnboardingHealthCheck` takes the readiness summary plus injectable label strings (so the pure helper is locale-agnostic and testable without i18n), and produces display-ready lines + a `ready` boolean. The `usable` branch renders it as a card; the `new`/`broken` branches don't (they go straight into the guided flow, which already names the missing piece). This keeps the three §9.1 branches visually distinct: usable = summary card, new/broken = guided steps.
- The Web `/api/stream` endpoint has no server-side follow-up queue: a concurrent send while the runner is running returns HTTP 409. The Desktop therefore implements the follow-up queue purely on the client — it keeps the composer enabled during a run, queues Enter-submitted messages locally, and auto-sends them one at a time after each turn completes. `selectSession` must distinguish a same-session reload (after a streamed turn) from an actual session switch so the queue/search state survives the post-turn transcript reload.
- Voice recording (§7.1) needs `getUserMedia`/`MediaRecorder` in the Tauri WKWebView plus an `NSMicrophoneUsageDescription` Info.plist key and a native media-capture permission delegate (`requestMediaCapturePermissionFor`). Whether Tauri 2 wires that delegate by default is unverified in this sandbox, so voice was deferred rather than shipped as an unverifiable half-feature.
- The existing `/api/settings/channel-instance` PUT is a full-instance upsert, not a partial patch: the desktop would have to round-trip agentId/credentials/sandboxEnabled it never receives (the credential-safe bootstrap deliberately omits them), so reusing it for an enable/rename toggle would wipe the server-owned fields. A narrow `/api/desktop/profiles` PATCH that accepts only `name`/`enabled` and preserves every other field server-side (`patchDesktopWebProfile`) is the safe contract for a credential-blind Desktop client. `sanitizeSettings` shallow-merges top-level keys, but `sanitizeChannels` starts from `current` and overlays only the channels present in the patch, so passing `channels: { web: { instances } }` leaves Telegram/Feishu/QQ/Weixin untouched — the narrow endpoint does not need to reconstruct the whole `channels` map.
- Desktop Settings mapper credential/path invariant (§11): a focused review of all 14 `src/lib/server/app/desktop*.ts` mappers (Providers, MCP, Agents, Skills, Host Bash, Sandbox, Tasks, Run History, Usage, Trace, plus Models/Profiles/Bootstrap/Handshake) confirmed every one projects fields explicitly and never spreads the raw source object. Secrets become booleans/counts (`apiKey`→`hasApiKey`; MCP `env`/`headers`/`args`→counts; Host Bash `envAllowlist`→count), absolute paths are dropped (`envFilePath`, `filePath`, `baseDir`, `cwd`, `dataRoot`, workspace/draft paths), and sensitive content is dropped (task `text`/prompts, `finalText`, raw Host Bash `command`, raw fact records). **Rule for any future desktop mapper: build the Desktop view field-by-field from the source and never `return {...source}` — a spread is the leak vector this boundary exists to prevent.** No leak was found; the WebView never receives a provider key, auth token, env var name/value, or absolute on-disk path. The review also found `desktopAgents.test.ts` was the only mapper test without a leak-absence assertion (its mapper drops the agent's system prompt and surfaces model-routing keys only as a count); a regression test now injects a system prompt, an agent-level key, and a raw routing key and asserts none reach the serialized `DesktopAgentItem`. As of this pass, every `desktop*.ts` mapper has executable leak-absence coverage, so the §11 invariant is test-enforced rather than only documented.

# 2026-06-28 — Onboarding launch-at-login slice

- Product/design brief is already fixed by plan §9.2 and `DESIGN.md`: add a fully functional launch-at-login choice inside the existing onboarding card, default off, with no visual redesign. Product Design saved context is absent, so repository sources remain authoritative.
- The app already owns autostart through the official Tauri plugin and exposes `set_login_start`; Settings General already toggles it and browser preview intentionally mutates only local in-memory status.
